# UAV资源分配算法性能评估指南

## 评估指标体系

### 1. 主要性能指标（Primary Metrics）

#### 1.1 分组投递率（PDR - Packet Delivery Ratio）
```
PDR = 接收数据包数 / 发送数据包数 × 100%

目标：≥ 85%
评价：PDR越高，算法效果越好
```

**意义**: 反映网络可靠性，是最重要的QoS指标之一

**计算方法**:
- 节点级PDR：单个节点的接收/发送比
- 网络级PDR：所有节点的平均PDR
- 流级PDR：每条业务流的PDR

**优秀/良好/一般标准**:
- 优秀：PDR ≥ 90%
- 良好：85% ≤ PDR < 90%
- 一般：70% ≤ PDR < 85%
- 较差：PDR < 70%

#### 1.2 端到端时延（E2E Delay）
```
Delay = Σ(接收时间 - 发送时间) / 接收包数

目标：≤ 100 ms
评价：时延越低，算法效果越好
```

**意义**: 反映网络实时性，对实时业务至关重要

**分析维度**:
- 平均时延
- 最大时延（99th百分位）
- 时延抖动（标准差）

**优秀/良好/一般标准**:
- 优秀：Delay ≤ 50 ms
- 良好：50 ms < Delay ≤ 100 ms
- 一般：100 ms < Delay ≤ 200 ms
- 较差：Delay > 200 ms

#### 1.3 吞吐量（Throughput）
```
Throughput = 接收总字节数 × 8 / 仿真时长 (bps)

评价：吞吐量越高，算法效果越好
```

**意义**: 反映网络容量和资源利用效率

**分析维度**:
- 总吞吐量（所有节点总和）
- 平均吞吐量（单节点平均）
- 吞吐量公平性（标准差/均值）

---

### 2. 资源分配效率指标

#### 2.1 信道利用率
```
信道利用率 = 使用该信道的节点数 / 总节点数

评价：分布越均匀，算法效果越好
```

**衡量方法**:
- 负载均衡度：标准差越小越好
- 信道冲突率：同信道邻居节点对数
- 信道复用因子：总节点数 / 信道数

#### 2.2 功率效率
```
功率效率 = 吞吐量 / 平均发射功率

评价：效率越高，算法效果越好
```

**意义**: 反映能量利用效率，对续航要求高的场景重要

#### 2.3 频谱效率
```
频谱效率 = 吞吐量 / (信道数 × 信道带宽)

评价：效率越高，算法效果越好
```

---

### 3. 网络拓扑指标

#### 3.1 网络连通性
```
连通性 = 活跃链路数 / 最大可能链路数 × 100%

评价：连通性越高越好，但也要考虑干扰
```

#### 3.2 平均节点度
```
平均度 = Σ(每个节点的邻居数) / 节点数

评价：适中为好（4-8为宜）
```

#### 3.3 链路质量分布
```
平均链路质量 = Σ(所有活跃链路的SINR) / 链路数

评价：SINR越高越好
```

---

### 4. 算法复杂度指标

#### 4.1 时间复杂度
- **Static**: O(N) - 最快
- **Greedy**: O(N²) - 较快
- **Graph Coloring**: O(N²) - 中等
- **Interference Aware**: O(N² × C) - 较慢

#### 4.2 计算开销
```
计算时间 = 资源分配函数执行时间

评价：时间越短越好，但要平衡性能
```

#### 4.3 重配置开销
```
重配置率 = 信道变更次数 / 重分配次数 × 100%

评价：过于频繁的重配置会影响稳定性
```

- 15个UAV，3个信道，500×500m²

- 30个UAV，4个信道，500×500m²

## 性能评估流程

### Step 1: 数据收集

运行仿真，收集原始数据：
```bash
./run_uav_simulation.sh graph_coloring 15 3 200
```

输出文件：
- `resource_allocation.csv` - 资源分配记录
- `qos_performance.csv` - QoS性能数据
- `topology_evolution.csv` - 拓扑演化数据
- `summary.txt` - 性能摘要

### Step 2: 数据分析

使用分析脚本：
```bash
python3 analyze_performance.py output/uav_resource_allocation_xxx/
```

### Step 3: 可视化对比

生成对比图表：
```bash
python3 compare_algorithms.py output/strategy1/ output/strategy2/ output/strategy3/
```

### Step 4: 统计检验

进行显著性检验（t-test）：
- 零假设：算法A和算法B性能无显著差异
- 置信度：95%
- 重复次数：≥10次（不同随机种子）

### 三、性能分析

#### 3.1 PDR分析
- Graph Coloring达到87.2%，满足≥85%的QoS要求
- Interference Aware性能最好(89.1%)，但计算开销大
- Static不满足要求，仅72.3%

#### 3.2 时延分析
- 所有算法中，Interference Aware时延最低(85.3ms)
- Graph Coloring满足≤100ms要求
- Static超标，达145.2ms

#### 3.3 综合评价
**推荐算法**: Graph Coloring
- 理由：性能与开销平衡最好
- PDR满足要求(87.2%)
- 时延满足要求(92.5ms)
- 计算开销可接受(5.7ms)
- 信道冲突少(6个)

## 常见问题

### Q1: 如何判断算法是否满足QoS要求？
**A**: 检查两个关键指标：
- PDR ≥ 85% ✓
- 时延 ≤ 100 ms ✓

两者都满足才算达标。

### Q2: 如何对比算法优劣？
**A**: 综合考虑三个维度：
1. 性能（PDR、时延、吞吐量）
2. 效率（资源利用率、计算开销）
3. 鲁棒性（不同场景下的稳定性）

### Q3: 如何选择最适合的算法？
**A**: 根据应用场景：
- 计算受限 → Static/Greedy
- 通用场景 → Graph Coloring（推荐）
- 高QoS要求 → Interference Aware

### Q4: 实验要做多少次？
**A**: 
- 基础实验：每种算法1次
- 标准实验：每种算法3-5次（不同随机种子）
- 严格实验：每种算法10次以上（统计检验）
