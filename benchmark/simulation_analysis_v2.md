## ⚠️ 关键问题分析：仿真结果仍不理想

### 1️⃣ **PDR 几乎未受影响（仍然接近 100%）**

#### 问题描述：
- **Moderate PDR**: 0.9992 - 0.9998（目标 < 0.9）❌
- **Hard PDR**: 0.9997 - 0.9999（目标 < 0.8）❌

#### 对比修改前后：
| 难度 | 修改前 PDR | 修改后 PDR | 变化 |
|------|-----------|-----------|------|
| Moderate | 0.9971-0.9998 | 0.9992-0.9998 | **无显著变化** ⚠️ |
| Hard | 0.9753-0.9999 | 0.9997-0.9999 | **反而提高了** ❌ |

#### 原因分析：
**统一通信距离到 200m 的副作用超过了增强干扰的效果！**

1. **通信距离增加导致链路质量提升**：
   - Moderate 从 80m → 200m
   - Hard 从 50m → 200m
   - **在较短距离下，链路更密集，信号强度更高，反而减少了丢包**

2. **干扰增强不足以抵消距离增加的好处**：
   - 虽然干扰节点数量增加（Moderate: 5→8, Hard: 12→16）
   - 但干扰是广播式的，在 200m 范围内，UAV 节点仍能找到足够强的信号路径

3. **MAC层重传机制仍然有效**：
   - Moderate 仍有 1 次重传
   - Hard 虽然 0 重传，但短距离下首次传输成功率很高

---

### 2️⃣ **吞吐量呈现异常模式**

#### 问题描述：
- **Easy**: 0.012-0.013 Mbps（极低，符合预期）✅
- **Moderate**: 0.57-0.60 Mbps（中等，符合预期）✅
- **Hard**: **12.97-15.21 Mbps**（极高，**严重异常**）❌

#### 异常原因分析：
**Hard 模式吞吐量是 Moderate 的 20-25 倍，这与高负载设计矛盾！**

1. **通信距离统一后，Hard 模式触发了更多成功传输**：
   - `commTriggerProb = 0.8`（极高触发频率）
   - `maxCommDistance = 200m`（远大于之前的 50m）
   - **结果**：几乎所有节点都能成功通信，导致吞吐量爆炸

2. **业务负载配置失衡**：
   - Hard: `heartbeatRate=8.0`, `videoDataRate=4.0`, `sensorDataRate=3.0`
   - 总计 **7Mbps 业务负载** + **极高触发频率**
   - **PDR 接近 100%** → 所有数据都成功传输 → 吞吐量极高

3. **干扰未能有效阻止传输**：
   - 16 个干扰节点 @ 6Mbps × 95% = 91.2 Mbps 理论干扰
   - 但在 200m 距离下，UAV 间距缩短，仍能找到低干扰时隙

---

### 3️⃣ **拓扑稳定性呈现正向增长（违反预期）**

| 难度 | 平均拓扑稳定性 | 预期 |
|------|--------------|------|
| Easy | 0.0267 | 低（因为长距离，链路稀疏） ✅ |
| Moderate | 0.0737 | 应较低 ❌ |
| Hard | 0.6352 | **应最低，实际最高** ❌ |

#### 原因：
- **统一到 200m 后，Hard 模式的链路数量大幅增加**
- 虽然信道质量差，但链路数量多且持续存在
- **拓扑稳定性 = 链路相似度**，链路越多越稳定

---

## 🔍 根本原因总结

### ❌ **本次修改的致命缺陷**：

**将通信距离统一到 200m 完全抵消了难度梯度设计！**

| 修改项 | 预期效果 | 实际效果 |
|--------|---------|---------|
| 统一距离 200m | 公平比较 | **消除了距离压力，反而让困难模式更容易** |
| 增加干扰节点 | 提高丢包率 | **在短距离高密度下，干扰效果被链路冗余抵消** |
| 减少重传次数 | 降低可靠性 | **短距离下首次传输成功率高，重传意义不大** |

### ✅ **正确的设计思路应该是**：

**通信距离不应统一，而应作为难度梯度的核心手段之一！**

---

## 💡 推荐的解决方案

### 方案 A：**保持距离差异化 + 适度增强干扰**（推荐）

```cpp
// Easy
config.maxCommDistance = 200.0;   // 长距离，低密度
config.numInterferenceNodes = 0;

// Moderate
config.maxCommDistance = 100.0;   // 中等距离（不是 200）
config.numInterferenceNodes = 6;  // 适度干扰
config.maxRetries = 1;

// Hard
config.maxCommDistance = 60.0;    // 短距离，高密度（不是 200）
config.numInterferenceNodes = 10; // 强干扰（不是 16）
config.maxRetries = 0;
```

**核心理念**：
- ✅ 距离越短 → 链路密度越高 → 拥塞越严重 → PDR 下降
- ✅ 结合信道衰落 + 高业务负载 + 适度干扰
- ✅ 避免距离过短导致链路过于稳定（当前 Bug 的根源）

---

### 方案 B：**统一距离 + 强化其他压力源**（次选）

如果必须统一距离（如您的需求），则需要：

1. **大幅降低 Hard 模式的物理层性能**：
   ```cpp
   // Hard 模式
   txPower = 5.0;           // 极低发射功率（从 10 降至 5）
   rxSensitivity = -70.0;   // 极差接收灵敏度（从 -82 升至 -70）
   noiseFigure = 15.0;      // 极高噪声系数（从 12 升至 15）
   ```

2. **降低业务负载以避免吞吐量爆炸**：
   ```cpp
   // Hard 模式
   heartbeatRate = 4.0;     // 从 8.0 降至 4.0
   videoDataRate = 2.0;     // 从 4.0 降至 2.0
   sensorDataRate = 1.5;    // 从 3.0 降至 1.5
   commTriggerProb = 0.5;   // 从 0.8 降至 0.5
   ```

3. **增加信道时变性（快衰落）**：
   - 引入 Rayleigh/Rician 快衰落模型
   - 增加多径效应

---

### 方案 C：**混合策略**（最佳）

**不同队形使用不同距离策略**：

```cpp
// 根据队形动态调整距离
if (formation == "line" || formation == "v_formation") {
    // 线性队形：节点间距大，使用统一距离
    config.maxCommDistance = 200.0;
} else {
    // 紧凑队形（cross, triangle）：使用差异化距离
    config.maxCommDistance = (difficulty == EASY) ? 200.0 :
                            (difficulty == MODERATE) ? 120.0 : 70.0;
}
```

**优点**：
- ✅ 保留队形特性的影响
- ✅ 在不同场景下合理分配难度压力
- ✅ 避免一刀切导致的副作用
