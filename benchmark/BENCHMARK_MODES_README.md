# RTK Benchmark 三种模式设计说明

## 📋 概述

RTK Benchmark 设计了三种难度模式（Easy、Moderate、Hard），分别对应不同的现实应用场景和技术挑战。所有模式使用统一的 50m 通信距离，通过信道质量、RTK精度、WiFi配置、业务负载和干扰强度来体现难度差异。

---

## 🌟 设计理念

### 核心原则
1. **公平竞争**：统一通信距离（50m）确保所有模式在相同基础条件下竞争
2. **现实映射**：每种模式对应具体的现实应用场景
3. **渐进挑战**：从理想环境到极端恶劣环境，逐步增加技术挑战
4. **多维压力**：通过信道、定位、通信、业务、干扰等多个维度构建难度梯度

### 技术挑战维度
- **信道环境**：路径损耗、多径衰落
- **定位精度**：RTK噪声和漂移
- **通信协议**：WiFi速率控制、重传机制
- **业务负载**：数据传输量和频率
- **干扰环境**：同频干扰节点数量

## 🟢 Easy 模式：理想测试环境

### 🎯 现实场景对应
**应用场景**：
- 室内测试实验室
- 空旷的室外测试场
- 概念验证和算法原型测试
- 系统功能验证

**典型用例**：
- UAV编队算法的初期验证
- RTK定位系统的基准测试
- 通信协议的性能上限测试
- 教学演示和培训

### ⚙️ 参数配置详解

#### 信道模型（理想环境）
```cpp
pathLossExponent0 = 2.0;  // 自由空间传播
pathLossExponent1 = 2.1;  // 轻微环境反射
pathLossExponent2 = 2.2;  // 轻微多径效应
nakagamiM0 = 3.0;         // 极轻微衰落（接近AWGN）
nakagamiM1 = 2.5;         // 轻微衰落
nakagamiM2 = 2.0;         // 轻微多径衰落
```

**含义**：
- **路径损耗指数 2.0**：符合自由空间传播模型，信号衰减最慢
- **Nakagami m=3.0**：接近高斯噪声，几乎没有多径衰落
- **设计目的**：提供最佳的信道条件，验证算法的理论性能上限

#### RTK定位（高精度稳定）
```cpp
rtkNoiseStdDev = 0.01;     // 1cm 标准噪声
rtkDriftInterval = 0;      // 无周期性漂移
rtkDriftDuration = 0;      // 无漂移持续时间
rtkDriftMagnitude = 0;     // 无漂移幅度
```

**含义**：
- **1cm 噪声**：接近RTK的理论精度极限
- **无漂移**：模拟理想的多频GNSS接收机
- **设计目的**：消除定位误差对算法性能的影响

#### WiFi通信（最优配置）
```cpp
wifiStandard = "80211n";
channelWidth = 20;         // 20MHz 带宽
rateControl = "ConstantRate";
dataMode = "HtMcs7";       // 最高速率 65Mbps
maxRetries = 10;           // 充足重传机会
```

**含义**：
- **HtMcs7**：802.11n的最高调制编码方案，65Mbps理论速率
- **10次重传**：最大化传输可靠性
- **设计目的**：提供最佳通信条件，专注算法逻辑验证

#### 业务负载（极轻）
```cpp
heartbeatRate = 1.0;       // 1Hz 心跳频率
heartbeatSize = 200;       // 200字节心跳包
videoDataRate = 0.1;       // 100kbps 视频流
sensorDataRate = 0.01;     // 10kbps 传感器数据
```

**含义**：
- **总计 < 0.12Mbps**：极低带宽需求
- **设计目的**：避免网络拥塞，专注算法性能验证

#### 通信触发（低频）
```cpp
commTriggerProb = 0.3;     // 30% 触发概率
maxCommDistance = 50.0;    // 50m 最大通信距离
```

**含义**：
- **低触发频率**：减少不必要的通信开销
- **设计目的**：最小化网络负载，专注核心功能

#### 干扰环境（无干扰）
```cpp
enableInterference = false;
numInterferenceNodes = 0;
```

**含义**：
- **完全无干扰**：理想的无线环境
- **设计目的**：消除外部因素，专注算法本身

## 🟡 Moderate 模式：城市应用环境

### 🎯 现实场景对应
**应用场景**：
- 城市无人机配送
- 城市搜救任务
- 城市环境监测
- 智慧城市无人机服务

**典型用例**：
- 美团/京东的城市配送无人机
- 消防部门的城市搜救无人机
- 环保部门的大气监测无人机编队
- 交通部门的城市交通监控

### ⚙️ 参数配置详解

#### 信道模型（城市环境）
```cpp
pathLossExponent0 = 2.8;   // 城市自由空间
pathLossExponent1 = 3.5;   // 城市反射增强
pathLossExponent2 = 4.0;   // 城市多径严重
nakagamiM0 = 1.0;          // Rayleigh衰落
nakagamiM1 = 0.7;          // 比Rayleigh更差
nakagamiM2 = 0.5;          // 严重多径衰落
```

**含义**：
- **路径损耗指数 2.8-4.0**：模拟城市建筑反射和多径效应
- **Nakagami m=0.5-1.0**：严重的多径衰落，接近实际城市环境
- **设计目的**：反映城市环境中的信号传播挑战

#### RTK定位（周期性漂移）
```cpp
rtkNoiseStdDev = 0.08;     // 8cm 标准噪声
rtkDriftInterval = 15.0;   // 15秒周期漂移
rtkDriftDuration = 4.0;    // 持续4秒
rtkDriftMagnitude = 0.5;   // 50cm 漂移幅度
```

**含义**：
- **8cm 基础噪声**：城市多径环境下的RTK精度
- **周期性漂移**：模拟城市环境中的多径效应和信号遮挡
- **设计目的**：反映城市RTK定位的实际挑战

#### WiFi通信（限制重传）
```cpp
rateControl = "ConstantRate";
dataMode = "HtMcs5";       // 中等速率 52Mbps
maxRetries = 1;            // 仅1次重传
```

**含义**：
- **HtMcs5**：降低速率以提高可靠性
- **1次重传**：限制重传次数，增加丢包压力
- **设计目的**：模拟城市环境中的通信约束

#### 业务负载（中等）
```cpp
heartbeatRate = 3.0;       // 3Hz 心跳频率
heartbeatSize = 600;       // 600字节心跳包
videoDataRate = 2.0;       // 2Mbps 视频流
sensorDataRate = 0.8;      // 800kbps 传感器数据
```

**含义**：
- **总计 ≈ 2.8Mbps**：中等业务负载
- **3Hz心跳**：提高编队协调频率
- **设计目的**：模拟实际应用的通信需求

#### 通信触发（中频）
```cpp
commTriggerProb = 0.6;     // 60% 触发概率
```

**含义**：
- **中等触发频率**：平衡性能和可靠性
- **设计目的**：反映城市应用的通信密度需求

#### 干扰环境（中等干扰）
```cpp
enableInterference = true;
numInterferenceNodes = 8;   // 8个干扰节点
// 干扰参数：4Mbps, 70%占空比, 1300字节包
```

**含义**：
- **8个干扰节点**：模拟城市WiFi环境
- **4Mbps干扰**：模拟其他WiFi设备的干扰
- **设计目的**：反映城市2.4GHz频段的拥塞情况

## 🔴 Hard 模式：极端恶劣环境

### 🎯 现实场景对应
**应用场景**：
- 军事作战环境
- 灾难救援现场
- 极地/海洋科考
- 复杂电磁环境

**典型用例**：
- 军用无人机编队作战
- 地震/海啸后的搜救任务
- 极地科学考察无人机
- 电磁干扰环境下的应急通信

### ⚙️ 参数配置详解

#### 信道模型（极端恶劣）
```cpp
pathLossExponent0 = 3.5;   // 极高路径损耗
pathLossExponent1 = 4.2;   // 严重反射损耗
pathLossExponent2 = 5.0;   // 极度多径损耗
nakagamiM0 = 0.5;          // 极严重衰落
nakagamiM1 = 0.3;          // 远差于Rayleigh
nakagamiM2 = 0.2;          // 极度恶劣环境
```

**含义**：
- **路径损耗指数 3.5-5.0**：极端环境下的信号衰减
- **Nakagami m=0.2-0.5**：极其严重的多径衰落
- **设计目的**：模拟最恶劣的无线传播环境

#### RTK定位（频繁大幅漂移）
```cpp
rtkNoiseStdDev = 0.2;      // 20cm 标准噪声
rtkDriftInterval = 8.0;    // 8秒周期，很频繁
rtkDriftDuration = 6.0;    // 持续6秒
rtkDriftMagnitude = 1.0;   // 1米大幅漂移
```

**含义**：
- **20cm 基础噪声**：恶劣环境下的RTK精度极限
- **频繁大幅漂移**：模拟极端环境中的信号遮挡和干扰
- **设计目的**：测试算法在极端定位条件下的鲁棒性

#### WiFi通信（无重传）
```cpp
rateControl = "ConstantRate";
dataMode = "HtMcs7";       // 最高速率，最易出错
maxRetries = 0;            // 无重传！一次失败即丢包
```

**含义**：
- **最高速率**：最大化误码率
- **0次重传**：一次失败即丢包，极端可靠性要求
- **设计目的**：模拟极端环境下的通信约束

#### 业务负载（超高）
```cpp
heartbeatRate = 8.0;       // 8Hz 极高频心跳
heartbeatSize = 1000;      // 1000字节大心跳包
videoDataRate = 4.0;       // 4Mbps 视频流
sensorDataRate = 3.0;      // 3Mbps 传感器数据
```

**含义**：
- **总计 ≈ 7Mbps**：超高业务负载
- **8Hz心跳**：极高频编队协调
- **设计目的**：模拟极端应用场景的高数据需求

#### 通信触发（高频）
```cpp
commTriggerProb = 0.8;     // 80% 极高触发概率
```

**含义**：
- **极高触发频率**：最大化网络负载
- **设计目的**：模拟极端环境下的密集通信需求

#### 干扰环境（极强干扰）
```cpp
enableInterference = true;
numInterferenceNodes = 16;  // 16个干扰节点
// 干扰参数：6Mbps, 95%占空比, 1472字节包
```

**含义**：
- **16个干扰节点**：模拟极度拥挤的电磁环境
- **6Mbps干扰**：高强度同频干扰
- **95%占空比**：几乎持续干扰
- **设计目的**：模拟军事或灾难环境中的电磁对抗

## 🎯 设计哲学

### 渐进式挑战设计
1. **Easy → Moderate**：从理想环境到现实应用
   - 引入城市环境的信道挑战
   - 增加RTK定位的不稳定性
   - 限制WiFi重传机制
   - 提升业务负载和通信密度

2. **Moderate → Hard**：从现实应用到极端环境
   - 大幅恶化信道条件
   - 显著增加RTK漂移和噪声
   - 完全消除重传机制
   - 极大提升业务负载和干扰强度

### 多维压力平衡
- **信道压力**：通过路径损耗和多径衰落
- **定位压力**：通过RTK噪声和漂移
- **通信压力**：通过速率控制和重传限制
- **负载压力**：通过业务数据量和频率
- **干扰压力**：通过同频干扰节点

### 现实场景映射
- **Easy**：实验室测试 → 算法验证
- **Moderate**：城市应用 → 产品开发
- **Hard**：极端环境 → 军用/应急

## 📈 预期性能表现

### PDR (Packet Delivery Ratio) 目标
- **Easy**: ≈ 100% (理想环境)
- **Moderate**: < 90% (城市环境挑战)
- **Hard**: < 80% (极端环境挑战)

### 吞吐量特征
- **Easy**: 极低 (专注算法验证)
- **Moderate**: 中等 (平衡性能与可靠性)
- **Hard**: 高负载 (测试极限性能)

### 时延特征
- **Easy**: 最低 (理想条件)
- **Moderate**: 中等 (城市环境影响)
- **Hard**: 最高 (极端环境挑战)

### 拓扑稳定性
- **Easy**: 最高 (稳定环境)
- **Moderate**: 中等 (城市环境变化)
- **Hard**: 最低 (极端环境不稳定)

## 🔧 使用建议

### 算法开发阶段
1. **概念验证**：使用 Easy 模式验证算法正确性
2. **性能优化**：使用 Moderate 模式优化实际性能
3. **鲁棒性测试**：使用 Hard 模式测试极端环境适应性

### 应用部署阶段
1. **城市应用**：参考 Moderate 模式参数
2. **应急应用**：参考 Hard 模式参数
3. **测试验证**：使用 Easy 模式进行基准测试

### 研究分析阶段
1. **算法对比**：在相同模式下对比不同算法
2. **环境适应**：分析算法在不同模式下的性能变化
3. **参数调优**：基于模式特点进行参数优化

---

## 📝 总结

RTK Benchmark 通过精心设计的三种模式，为无人机编队通信算法提供了全面的测试环境：

- **Easy 模式**：提供理想的测试环境，专注算法逻辑验证
- **Moderate 模式**：模拟现实城市应用，平衡性能与可靠性
- **Hard 模式**：模拟极端恶劣环境，测试算法鲁棒性

通过统一通信距离（50m）和多维压力设计，确保测试结果的公平性和可比性，为无人机编队通信技术的发展提供标准化的评估平台。
